Concept

Claude Codeは、チャットではなく作業者に近い。

公式ドキュメントでは、Claude Codeはコードベースを読み、ファイル編集、コマンド実行、開発ツール連携を行う agentic coding tool と説明されています。つまり、質問相手であると同時に、手元のプロジェクトで作業できる存在です。

読む

プロジェクトを理解する

ファイル構成、使っている技術、入口になるファイル、既存の実装パターンを読ませられます。

変える

ファイルを編集する

変更案を作り、承認後にファイルを書き換えます。初心者は小さい変更から依頼します。

確かめる

コマンドを実行する

テスト、ビルド、lint、Git確認などを実行できます。何を実行するかは許可画面で確認します。

最初の合言葉は「読んで、計画して、確認してから変更」

Claude Codeは強力なので、いきなり「直して」よりも、まず「変更せずに読んで」「計画を出して」「変更対象を説明して」と依頼する方が安全です。

Install

インストールとログイン。

公式Quickstartでは、macOS / Linux / WSL向けにNative Install、Homebrewなどの方法が案内されています。ここではMac利用者向けに、公式の代表的な方法だけを載せます。

Terminalで実行: Native Install
curl -fsSL https://claude.ai/install.sh | bash
Terminalで実行: Homebrewで入れる場合
brew install --cask claude-code
Terminalで実行: 起動してログインだけ確認する
claude
項目意味初心者向けメモ
Native Install公式の推奨インストール方法。公式ドキュメントでは自動更新されると説明されています。
HomebrewMacのパッケージ管理で入れる方法。公式ドキュメントではHomebrew版は自動更新ではないため、必要に応じて brew upgrade claude-code します。

/loginClaude Code内でログインし直すコマンド。アカウントを切り替える時や再認証する時に使います。

Claude Pro/Max/Team/Enterprise、Console、クラウドプロバイダ、会社のSSOなど、使えるログイン方法は組織によって違います。教材では操作を説明しますが、実際のアカウント種別は管理者の案内を優先します。

Practice Project

インストール直後は、練習用プロジェクトを作る。

Claude Codeは「プロジェクトの中」で使う道具です。インストールしただけでは、まだ読む対象も変更する対象もありません。まず小さな練習用フォルダを作り、そこをClaude Codeの作業場所にします。

Terminalで実行: 練習用プロジェクトを作る
mkdir -p ~/practice/claude-code-lab
cd ~/practice/claude-code-lab
cat > README.md <<'EOF'
# Claude Code Lab

Claude Codeの練習用プロジェクトです。
最初は小さなHTMLだけを変更します。
EOF

cat > index.html <<'EOF'
<!doctype html>
<html lang="ja">
<head>
  <meta charset="utf-8">
  <title>Claude Code Lab</title>
</head>
<body>
  <h1>Claude Code Lab</h1>
  <p>ここをClaude Codeで少しずつ改善します。</p>
</body>
</html>
EOF
Terminalで実行: Gitで変更を見られる状態にする
git init
git add README.md index.html
git commit -m "Create Claude Code practice project"
なぜGitまで準備するのか

Claude Codeにファイルを変更させた後、git status と git diff で「何が変わったか」を確認するためです。AIに任せる時ほど、差分を見られる状態にしておくと安心です。

commitで名前やメールのエラーが出た場合

Gitのユーザー名やメールが未設定です。先にGit教材の「Author identity unknown」の説明を確認してください。commitできない場合でも、練習用ファイル自体は作成されています。

First Prompts

最初は、変更させずに読ませる。

公式Quickstartでも、最初の質問として「このプロジェクトは何をするものか」「技術構成は何か」「入口はどこか」「フォルダ構成を説明して」といった読み取り系の質問が紹介されています。

Claude Code内で入力: まず全体像を聞く
変更はせずに、このプロジェクトの目的、主要なファイル、フォルダ構成を説明してください。
Claude Code内で入力: 技術構成を聞く
このプロジェクトで使っている技術、起動方法、テスト方法を調べて説明してください。まだファイルは変更しないでください。
Claude Code内で入力: 作業計画だけ出させる
この修正を行う場合、どのファイルを変更する必要がありそうか、作業計画だけを出してください。まだ編集はしないでください。
依頼文に「まだ変更しないでください」を入れる

Claude Codeの権限設定とは別に、会話上の指示として「まず調査」「計画だけ」と伝えると、作業の進め方が安定します。大きな変更ほど、最初に計画を出させます。

Permissions

許可画面は、怖がる場所ではなく確認する場所。

公式ドキュメントでは、Claude Codeの権限は読み取り、コマンド実行、ファイル変更などで段階化されています。初心者は、何を許可しているのかを日本語に置き換えて読むことから始めます。

種類公式上の考え方初心者の見方
Read-onlyファイル読み取りやGrep。通常、承認不要。読んで説明するだけ。変更はしない。
Bash commandsシェルコマンド実行。承認が必要。npm test や git status など、実行内容を見て許可する。
File modificationファイル編集や書き込み。承認が必要。どのファイルをどう変えるのかを確認する。
/permissions権限ルールを確認・管理する画面。許可した内容が分からなくなった時に見る。
Claude Code内で入力: 権限を確認する
/permissions
モード意味この教材での扱い
default標準。最初のツール利用時に許可を求める。通常はここから始める。
planファイル読み取りや読み取り系コマンドで調査し、編集はしない。初回や大きな変更前におすすめ。
acceptEdits作業ディレクトリ内の編集や一部のファイル操作を自動承認。慣れてから使う。
auto安全チェック付きで自動承認する研究プレビュー。組織方針を確認してから使う。
bypassPermissions多くの許可確認を省略する。初心者は使わない。公式もコンテナやVMなど隔離環境での利用を前提に注意しています。
Terminalで実行: Plan modeで起動する
claude --permission-mode plan
許可してよいか迷ったら、一度止める

rm、sudo、git reset、git push、外部サービスへのアクセス、よく分からない長いコマンドは、すぐ許可しません。「このコマンドは何をしますか」とClaude Codeに説明させてから判断します。

Project Memory

毎回説明することは、CLAUDE.mdに置く。

公式ドキュメントでは、CLAUDE.md はClaude Codeに永続的な指示を与えるMarkdownファイルとして説明されています。毎回チャットで説明しているルール、ビルド方法、テスト方法、注意点を、あらかじめファイルに書いておく仕組みです。

自分の共通ルール

~/.claude/CLAUDE.md

自分がどのプロジェクトでも守ってほしい好みや作業ルールを書きます。例: 日本語で回答する、変更前に方針を説明する、勝手にcommitしない。

プロジェクト共有

./CLAUDE.md

プロジェクト全員に必要なルール、起動コマンド、テストコマンド、コーディング規約を書きます。Gitで共有する対象です。

プロジェクト内の個人用

./CLAUDE.local.md

自分だけのURL、検証環境、個人的な好みなどを書きます。公式では .gitignore に入れる使い方が説明されています。

場所影響範囲Gitで共有するか書く内容
~/.claude/CLAUDE.md自分の全プロジェクト通常は共有しない自分の作業スタイル、返答言語、確認してほしいこと。
./CLAUDE.md今のプロジェクト共有するチーム全員に必要なプロジェクト固有のルール。
./.claude/CLAUDE.md今のプロジェクト共有する./CLAUDE.md と同じく、プロジェクト指示として使える置き場所。
./CLAUDE.local.md今のプロジェクト、自分だけ共有しない個人の検証URL、ローカル環境、秘密ではないが共有不要な情報。
読み込まれる指示は、階層ごとに役割が違う

~/.claude/CLAUDE.md は「自分はいつもこう進めたい」という個人設定です。./CLAUDE.md は「このプロジェクトではこう進める」という共有設定です。両方ある場合、Claude Codeはそれぞれを文脈として読みます。

Claude Code内で入力: CLAUDE.mdのたたき台を作る
/init
/init で作られる内容は、プロジェクトによって変わる

公式ドキュメントでは、/init はコードベースを分析して、ビルドコマンド、テスト手順、プロジェクトの慣習などを含む CLAUDE.md のたたき台を作るものとして説明されています。

下の例は、この教材用に用意したサンプルです。/init を実行した時に、必ずこの内容がそのまま作られるわけではありません。

書くとよいこと

毎回説明している事実

起動コマンド、テストコマンド、ファイル構成、命名ルール、デザイン方針、禁止したい作業、レビュー観点を書きます。

避けたいこと

長すぎる願望リスト

曖昧な精神論、古い情報、矛盾したルール、秘密情報、APIキー、パスワードは書きません。短く、具体的にします。

~/.claude/CLAUDE.mdの例: 個人の共通ルール
# My Claude Code Preferences

## Communication
- Reply in Japanese unless I ask otherwise.
- Explain the plan before making large changes.
- Keep summaries concise and practical.

## Safety
- Do not commit or push unless I explicitly ask.
- Do not run destructive commands without explaining the risk first.
- When unsure, ask a short question before proceeding.

CLAUDE.mdは命令書だが、強制装置ではない

公式ドキュメントでは、CLAUDE.mdはコンテキストとして読み込まれるもので、権限を強制する仕組みではないと説明されています。危険な操作を止めるには、権限設定や許可画面の確認が必要です。

秘密情報は書かない

CLAUDE.md はClaude Codeが毎回読むための作業メモです。APIキー、パスワード、個人情報、社外秘の接続情報は書きません。必要な場合でも、秘密情報そのものではなく「環境変数 API_KEY を使う」のように名前だけを書きます。

Git Guardrail

AIに任せても、差分は自分で見る。

公式Quickstartでは、Claude CodeにGit操作を会話で依頼できることが紹介されています。ただし学習段階では、commitやpushまで自動で進めず、まず差分確認を自分で行うのが安全です。

Terminalで実行: 作業前の確認
git status
git branch
Terminalで実行: 作業後の確認
git status
git diff
Claude Code内で入力: Git確認を依頼する
変更内容を説明してください。まだcommitもpushもしないでください。

作業前に git status

未保存の変更や、別作業の途中でないかを見ます。

Claude Codeに小さく依頼

1回の依頼で、1つの修正、1つのページ、1つの目的に絞ります。

作業後に git diff

AIの説明だけで判断せず、変更された行を確認します。

納得してからcommit

commitやpushは、意味が分かった後に行います。
