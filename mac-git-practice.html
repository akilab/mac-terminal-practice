Concept

GitとGitHubを分けて考える

Gitは手元で履歴を作るコマンド、GitHubはリモートリポジトリを共有するWebサービスです。

Git

手元の履歴管理

変更をcommitとして保存し、過去の状態を追えるようにします。

GitHub

共有とレビュー

pushされたブランチを見て、Pull Requestで取り込みます。

Practice

自分のHTMLだけ編集

今回は members/ の自分用ファイルだけを編集します。

今回の作業の流れ

共有元の main を直接触らず、自分用ブランチで変更を作り、そのブランチをGitHubへ送ります。その後、GitHub上でPull Requestを作り、管理者に main へ取り込んでもらいます。

Clone

リポジトリを手元にコピーする

最初は ~/practice に練習リポジトリをcloneします。

cloneする
mkdir -p ~/practice
cd ~/practice
git clone https://github.com/akilab/mac-git-practice-lab.git
cd mac-git-practice-lab
状態を確認する
git status
git remote -v
git log --oneline
origin とは何か

git clone すると、clone元のGitHubリポジトリに origin という名前が自動で付きます。つまり origin は「GitHub上のこの練習リポジトリ」を指す短い名前です。確認するには git remote -v を使います。

Branch

作業ブランチを作る

main で直接作業せず、自分用のブランチを作ります。

名前別のブランチ例
git switch -c feature/update-akr
# または
git switch -c feature/update-rui
# または
git switch -c feature/update-ayu
# または
git switch -c feature/update-rie
ブランチ名は作業内容が分かる名前にする

feature/update-akr のように、何をするブランチか分かる名前にします。

switch

git switch -c の意味

switch はブランチを切り替えるコマンドです。-c を付けると、新しいブランチを作成して、そのブランチへ切り替えます。

feature/

必須ではなく命名ルール

feature/ は必須ではありません。git switch -c update-akr でも動きます。ただし、作業用ブランチだと分かりやすくするために feature/ を付ける現場が多いです。

checkoutで同じことをする場合
git checkout -b feature/update-akr
checkout でもブランチは作れます

昔からよく使われていた形は git checkout -b ブランチ名 です。現在は、ブランチ切り替えには git switch、ファイルを戻す操作には git restore と役割を分けて説明されることが増えています。

Stage

commitする前に、変更を選ぶ

Gitには、作業中の変更をいきなりcommitせず、いったん「commit予定」に入れる場所があります。これがstageです。

Working Tree

作業中の場所

エディタでファイルを編集した状態です。まだ履歴には保存されていません。

Staging Area

commit予定の箱

git add した変更だけが入ります。次のcommitに含めるものを選ぶ場所です。

Repository

履歴として保存

git commit すると、stageに入っている変更が履歴として保存されます。

stageの状態を確認する
git status
git diff
git add members/akr.html
git status
git diff --staged
commitとpushの違い

commit は手元のリポジトリに履歴を作る操作です。まだGitHubには送られていません。push は、その手元のcommitをGitHubへ送る操作です。

なぜstageがあるのか

作業中の変更の中から「今回のcommitに入れるもの」だけを選べるようにするためです。たとえば、HTMLの回答欄だけをcommitし、メモ書きや一時ファイルは含めない、という判断ができます。

Commit

変更を履歴にする

git add でstageに入れ、git commit でstage内の変更を履歴として保存します。

addしてcommitする
git add members/akr.html
git diff --staged
git commit -m "Update akr profile"
git log --oneline
git add . は慣れてから

git add . は、今いるディレクトリ以下の変更をまとめてcommit対象に入れるコマンドです。便利ですが、不要な一時ファイル、別の人のファイル、まだ確認していない変更まで入ることがあります。最初は git add members/akr.html のようにファイル名を指定し、「何をcommitするのか」を自分で確認する練習にします。

add前後で何が変わるか見る
git diff
git add members/akr.html
git diff --staged
git status
Push

GitHubへ送る

手元のブランチをGitHubへpushします。初回は -u を付けます。

pushする
git push -u origin feature/update-akr

ブランチ名を自分のものに変えて実行してください。

push

commitを送る

git push は、手元のcommitをリモートリポジトリへ送ります。pushしただけでは、まだ main には取り込まれません。

origin

送り先の名前

origin はclone元のGitHubリポジトリについた名前です。つまり「GitHub上の練習リポジトリへ送る」という意味です。

-u

次回の省略設定

-u は upstream の設定です。手元のブランチとGitHub上のブランチを対応付け、次回から git push や git pull だけで済むようにします。

pushとPull Requestは別の操作

git push -u origin feature/update-akr は、作業ブランチをGitHubへ送る操作です。Pull Requestは、その後にGitHubの画面で「このブランチの変更を main に取り込んでください」と依頼する操作です。

Pull Request

取り込みを依頼する

push後、GitHub画面でPull Requestを作ります。

Pull Request説明文の例
## 変更内容
- members/akr.html の回答欄を更新しました

## 練習したこと
- clone
- branch
- diff
- add
- commit
- push
- Pull Request
After Merge

取り込まれた後に手元を更新する

Pull Requestが取り込まれたら、手元の main を最新にします。

mainを更新する
git switch main
git pull
git branch -d feature/update-akr
main

共有される本線

main はリポジトリの基準になるブランチです。自分の作業ブランチは、main から分かれて作業するための場所です。

switch main

本線へ戻る

git switch main は、今いる作業ブランチから main へ戻る操作です。ブランチは複数存在でき、必要に応じて切り替えられます。

pull

取り込み結果を受け取る

Pull Requestが取り込まれると、GitHub上の main が更新されます。git pull で、その更新を手元の main へ持ってきます。

git branch -d は作業済みブランチの片付け

git branch -d feature/update-akr は、手元に残っている作業ブランチを削除します。Pull Requestが取り込まれ、main に変更が入った後なら、その作業ブランチは役目を終えています。削除しても、commit自体は main に残ります。

なぜ main へ戻れるのか

ブランチは「別の作業場所」です。feature/update-akr で作業していても、main は消えていません。作業が終わったら git switch main で本線へ戻り、git pull で最新状態にします。

Troubleshooting

実習で詰まりやすいところ

WindowsとWSLを併用する場合、Gitそのものより「どのフォルダを見ているか」「どちらのGitで認証しているか」で詰まることがあります。

Folder

VSCodeとTerminalの場所を合わせる

VSCodeで編集しているフォルダと、Terminalで git status しているフォルダが違うと、git diff に変更が出ません。

Auth

WindowsとWSLの認証は別

Windows PowerShellではpushできても、WSLではGitHub認証が未設定でpushできないことがあります。

Author

commitには名前とメールが必要

Author identity unknown が出たら、Gitの user.name と user.email を設定します。

今いる場所とブランチを確認する
pwd
git rev-parse --show-toplevel
git branch --show-current
git status
commitの作者情報を設定する
git config --global user.name "akilab"
git config --global user.email "4317288+akilab@users.noreply.github.com"
git config --global user.name
git config --global user.email
GitHubのnoreplyメール

4317288+akilab@users.noreply.github.com のようなメールは、GitHubが用意する公開用メールです。commitをGitHubアカウントに紐づけつつ、本物のメールアドレスを公開しないために使います。

merge後のDelete branch

Pull Requestをmergeした後にGitHubで表示される Delete branch は、GitHub上の作業ブランチを片付ける操作です。変更内容は main に取り込まれているため、押して問題ありません。手元のブランチは別途 git branch -d feature/update-akr で削除します。
